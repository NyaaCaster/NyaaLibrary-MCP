## 1. 架构分析

### 1.1 结论：成立

- **两级 owner 结构对标现有「知识库→详情」** —— 与代码现状同构（`App.tsx` 两级路由、`KbDetailPage` tab 切换），复用面最大，判断正确。
- **运维端点走 `requireSession`、与猫猫成对 token 链路隔离** —— 已核对 `routes/index.ts`：`/memory` 挂在 `requireSession` 之前用自己的 fail-closed 鉴权，`/kb`、`/config`、`/`（library）挂在之后。新增 `/owners` 挂在 `requireSession` 之后是自洽的，前端 `api` 自动带管理员 HMAC 令牌，无需接触 `KB_SERVER_TOKEN`。隔离设计正确。

### 1.2 数据组织的一个隐含前提（需在 SSOT 标注）

`listOwners()` 只能列出**在 `memory_entries` 或 `profiles` 中出现过**的 owner。一个既无记忆、又无画像的 owner 在前端不可见——这是合理的（没有数据就没有卡片），但 SSOT 未明说。**建议**：在 §3.1 补一句"owner 列表由数据反查得出，非独立注册表"，避免实施者误以为存在 owner 主表。

## 3. 关键发现（按严重度）

### F1 — 越权级联删除的守卫顺序（严重度：高 · 正确性）

**问题**：SSOT §3.3 与 §4.1 写"事务删 `memory_entries` + `vec_mem` + `mem_fts`，`WHERE id=? AND owner_key=?`"。但后两张表没有 `owner_key`。若照字面实现，要么 SQL 报错，要么实施者退化成"三表都按 rowid 删"——那样 §3.3 声称的越权防护就**失效了**：路径里传入错误的 `ownerKey`，`memory_entries` 那条按双条件删不中（0 行），但 `vec_mem`/`mem_fts` 仍会按 rowid 被删掉，造成孤儿向量/索引被误删、且 owner 隔离形同虚设。

**修正（写入 SSOT）**：`deleteMemoryById(ownerKey, id)` 事务内**严格顺序**：

```
1. const info = DELETE FROM memory_entries WHERE id=? AND owner_key=?   // 双条件
2. if (info.changes === 0) return false;   // owner 不匹配 → 整体放弃，不碰另两表
3. DELETE FROM vec_mem  WHERE rowid=?   // 仅在步骤1命中后，按 id 级联（守卫懒建）
4. DELETE FROM mem_fts  WHERE rowid=?
5. return true
```

owner 校验的唯一权威点是**步骤 1 的 changes**，另两表只做 rowid 级联。SSOT §3.3 / §4.1 的表述据此改写。

### F2 — 补充上传会触发向量表重建、销毁猫猫既有向量（严重度：中 · 破坏性副作用）

**问题**：`writeMemory` 内部调 `ensureMemVecTable(dim)`。若嵌入配置的维度与库中已记录的 `mem_vec_dim`（当前实例=1536）**不一致**，`ensureMemVecTable` 会 `DROP TABLE vec_mem` 后重建（`db/index.ts:164-167`）——猫猫已沉淀的向量行会被清空。这是 `writeMemory` 的既有行为，但本次改造**新开了一个"人点按钮就触发"的入口**，把这个破坏性副作用暴露给了管理员，且此前有"嵌入配置是否正确运行未验证"的前提（SSOT §1.3）。

**处置建议（择一，写入 SSOT 风险节 + P2 后端实现约束）**：
- 最小改动：补充上传弹窗提交前，前端明确提示"写入将使用当前嵌入配置；若维度与既有记忆不一致，既有向量索引会重建"，让人知情。
- 或（更稳）：P2 在运维 POST 端点内先比对 `getMemVecDim()` 与当前配置维度，不一致时**拒绝写入并返回明确错误**，不静默重建。**推荐此项**——运维台不应成为静默毁数据的入口。

> 注：此项与 SSOT §1.3"不修嵌入配置"不冲突——这里不是修配置，是防止运维写入**误伤既有数据**。

### F3 — 画像编辑只能增改键、无法删单键（严重度：中 · 能力缺口）

**问题**：`upsertProfile` 用 `{...old, ...patch}` 字段级 merge（`services/memory.ts:53`）。经 PUT `profile_patch` 只能**新增或覆盖**键，**无法删除** `profile_json` 中的单个键。SSOT §2.1 目标写"删除用户画像"，实际只能整条删（`deleteProfile`），做不到"删画像里的某一个字段"。

**处置（写入 SSOT，明确边界）**：二选一并在文档定性——
- (a) 维持现状：画像编辑=整条 JSON 增改 + 整条删除，**不支持删单键**。列为已知限制。（改动小，推荐 V2 先这样）
- (b) 若要支持删单键：需前端做"整条 JSON 覆盖式编辑"，且后端另加一个"整条替换"语义端点（`PUT` 覆盖而非 merge），不能复用 `upsertProfile` 的 merge。改动较大。

SSOT 目前默认读者以为 (b) 可用，属**表述性缺陷**，必须点明取哪条。
